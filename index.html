GenAI Engineering Reference Guide

OpenAI SDK · LangChain · LangGraph · FastAPI · ChromaDB · LlamaIndex · Transformers · PEFT

This guide maps every function and parameter from eight library cheat-sheets back to a real production system: ICPA (Insurance Claims Processing Automation), a three-tier LangGraph pipeline processing roughly 50,000 German insurance emails a day at Vodafone. Tier 1 is a TF-IDF fast path handling about 60% of traffic, Tier 2 is ChromaDB-backed RAG with semantic caching, and Tier 3 is an LLM fallback for the hard cases. RAGAS faithfulness moved from 72% to 91% through MMR retrieval, metadata filtering, and tighter prompt grounding; LLM escalation dropped from 40% to 12%; the Redis semantic cache held an ~85% hit rate.

Each section covers what the library is for, every function/parameter from the source sheet with a one-line explanation, a worked code sample wired to ICPA-style data, and notes on where it shows up in interview narratives or the Go/GCP transition at SanDisk.

1. OpenAI SDK

Used for: accessing GPT models, embeddings, image generation, tool calling, and structured outputs.

In ICPA, the OpenAI SDK call pattern (or its Ollama-compatible equivalent) is what TIER 3 falls back to when Tier 1's TF-IDF path and Tier 2's RAG path both fail to produce a confident answer — that's the ~12% of traffic that reaches an LLM at all after optimization. The shape of every call below is identical whether you're hitting OpenAI's hosted API or a local llama3.2 server through an OpenAI-compatible client, which is why this SDK is worth knowing cold: it's the lingua franca every other framework (LangChain, LlamaIndex) wraps.

Worked example — Tier 3 fallback classification with structured output

This mirrors the actual escalation path: a claim email that TF-IDF and RAG couldn't confidently classify gets one structured call, with temperature pinned to 0 and a JSON schema so the output is parseable without a fragile regex.

python
from openai import OpenAI
from pydantic import BaseModel

client = OpenAI()  # reads OPENAI_API_KEY, or point base_url at Ollama

class ClaimDecision(BaseModel):
    category: str          # e.g. 'auto_collision', 'health_reimbursement'
    confidence: float
    requires_human_review: bool
    reasoning: str

def classify_claim_tier3(email_text: str, retrieved_context: str) -> ClaimDecision:
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        temperature=0,                # deterministic — classification, not creative writing
        max_tokens=300,
        response_format={"type": "json_schema",
                          "json_schema": {"name": "claim_decision",
                                          "schema": ClaimDecision.model_json_schema()}},
        messages=[
            {"role": "system", "content": "You triage German insurance claim emails."},
            {"role": "user", "content": f"Context:\n{retrieved_context}\n\nEmail:\n{email_text}"},
        ],
    )
    raw = response.choices[0].message.content
    print(response.usage)          # token accounting -> per-tenant cost logging
    return ClaimDecision.model_validate_json(raw)

Streaming variant (any user-facing chat surface)

stream = client.chat.completions.create(
    model="gpt-4o-mini",
    stream=True,
    messages=[{"role": "user", "content": "Summarize this claim's status in plain German."}],
)
for chunk in stream:
    delta = chunk.choices[0].delta.content
    if delta:
        print(delta, end="", flush=True)

Embeddings — the bridge into ChromaDB

emb = client.embeddings.create(
    model="text-embedding-3-small",
    input=["Schadenmeldung: Glasbruch am 14.03."],
)
vector = emb.data[0].embedding   # this is what .add() in ChromaDB stores
Interview / transition notes
  • You actually used Cohere multilingual embeddings in production, not OpenAI's — be ready to explain why (German-language coverage, cost) if asked to compare to embeddings.create().
  • response_format with a Pydantic-derived schema is the cleanest answer to "how do you guarantee structured output from an LLM" — better than prompting nicely and regexing the response.
  • In Go at SanDisk, tool_choice/response_format map to typed structs you unmarshal into; the schema-first discipline carries over even though the SDK syntax doesn't.

2. LangChain

Used for: building LLM applications, RAG pipelines, prompts, chains, and agents.

LangChain sits below LangGraph in the stack — LangGraph orchestrates the overall agent loop, but the individual nodes inside that graph (the retrieval step, the prompt-building step, the LLM call) are frequently LangChain Runnables under the hood. The LCEL pipe operator (|) is worth being fluent in, since it's the cleanest way to describe "retrieve, then stuff into prompt, then call model, then parse" in a single composable expression — exactly what TIER 2 RAG does before LangGraph hands control to it.

Function & parameter reference

ChatAnthropic()Same idea, pointed at Claude — useful when comparing provider behavior side by side.

Worked example — LCEL retrieval chain feeding Tier 2

The chain form of what ICPA's Tier 2 does: hybrid retrieval result gets stuffed into a grounded prompt, the model answers, and the parser strips it down to plain text before returning up to the LangGraph node that called it.

from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough, RunnableParallel

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

prompt = ChatPromptTemplate.from_messages([
    ("system", "Answer using ONLY the provided claim context. German insurance domain."),
    MessagesPlaceholder("chat_history", optional=True),
    ("human", "Context:\n{context}\n\nQuestion:\n{question}"),
])

def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)

rag_chain = (
    RunnableParallel(
        context=retriever | format_docs,   # hybrid BM25+dense retriever
        question=RunnablePassthrough(),
    )
    | prompt
    | llm
    | StrOutputParser()
)

answer = rag_chain.invoke("Welche Unterlagen fehlen fuer Schadensfall #4471?")

Structured output for downstream routing

from pydantic import BaseModel

class EscalationCheck(BaseModel):
    needs_llm_fallback: bool
    missing_fields: list[str]

structured_llm = llm.with_structured_output(EscalationCheck)
result = structured_llm.invoke("Does this claim have all required fields? ...")
if result.needs_llm_fallback:
    route_to_tier3(claim)

Interview / transition notes
  • Be precise about LangChain vs. LangGraph: LangChain composes a single linear-ish pipeline; LangGraph adds state, cycles, and conditional branching. ICPA needed LangGraph because Tier 1 → 2 → 3 is a conditional, stateful escalation path, not a fixed chain.
  • with_structured_output() and PydanticOutputParser() solve the same problem two ways — know that the former is preferred now since it uses native tool-calling rather than prompting for parseable text.
  • RunnableParallel is the direct analog of Go's goroutines + sync.WaitGroup fan-out/fan-in pattern — a useful bridge when explaining concurrency to a Go-focused interviewer.
